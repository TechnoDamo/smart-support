# Hybrid Retrieval и ограничения текущего BM25 sparse path

Документ фиксирует, как сейчас работает гибридный retrieval и почему текущий BM25 sparse path нужно считать промежуточным решением, а не финальной production-архитектурой.

## 1. Как retrieval устроен сейчас

Система использует два сигнала:

- `dense` — плотные эмбеддинги от embedding-провайдера
- `sparse` — BM25-подобный sparse-вектор, который строится локально в Python

Поиск выполняется так:

1. query превращается в dense и sparse представление
2. Qdrant получает отдельный dense search и sparse search
3. результаты объединяются через Reciprocal Rank Fusion (RRF)
4. итоговый top-k возвращается в приложение

## 2. Что важно понимать про dense

Dense-размер фиксирован и задаётся явно:

```env
EMBEDDING_VECTOR_SIZE=1536
```

Именно этот размер используется при создании dense-конфигурации коллекции в Qdrant.

## 3. Что важно понимать про sparse

Sparse-часть **не имеет фиксированной размерности**, как dense.

Вместо этого используется словарь токенов `vocab`, где:

- токен получает индекс в момент первого появления
- sparse-вектор — это просто пары `(indices, values)`
- фактическая «размерность» равна текущему размеру словаря

То есть sparse space сейчас зависит от того, какие токены уже были встречены процессом.

## 4. Точные ограничения текущей реализации

### 4.1 Словарь живёт только в памяти процесса

После рестарта backend-а словарь, `df`, `total_docs`, `avg_doc_len` теряются.

Следствие:

- sparse-пространство не является устойчивым между перезапусками.

### 4.2 Индексы токенов зависят от порядка первого появления

Если токены были встречены в другом порядке, индексы в словаре будут другими.

Следствие:

- один и тот же токен может получить разный индекс после рестарта или на другой инстанс-машине.

### 4.3 Query-side rebuild может не совпасть с тем, что лежит в Qdrant

При пустом BM25 backend лениво достраивает sparse-статистику по уже сохранённым чанкам.
Но если порядок чтения корпуса другой, mapping токен→индекс может не совпасть с тем, который был при ingestion.

Следствие:

- sparse query может обращаться уже не к тем координатам, что были записаны в Qdrant.

### 4.4 Статистика корпуса меняется, а старые sparse-векторы не переиндексируются

BM25 зависит от corpus-level статистики:

- `df`
- `total_docs`
- `avg_doc_len`

Но старые sparse-векторы в Qdrant не пересчитываются каждый раз при добавлении новых документов.

Следствие:

- часть документов в Qdrant закодирована старыми весами, а новые query могут считаться с новыми весами.

### 4.5 Удаление документов не вычитает их из BM25-статистики в памяти

Документы и точки можно удалить, но in-memory BM25 state не умеет корректно «дедуплицировать назад» статистику.

Следствие:

- со временем sparse-статистика расходится с реальным живым корпусом.

### 4.6 Несколько backend-инстансов небезопасны для такой sparse-модели

Если у разных инстансов разный порядок построения vocab, они получат разные индексы токенов.

Следствие:

- multi-instance deployment потенциально ломает sparse retrieval.

### 4.7 Токенизация очень простая

Сейчас используется простой regex-токенизатор без:

- лемматизации
- stemming
- stopword-анализа
- языко-специфических правил
- phrase handling

Для русского языка это особенно чувствительно.

## 5. Почему это ещё терпимо на MVP

В одном процессе, на небольшом корпусе и без частых рестартов такой подход может работать приемлемо как временный pragmatic MVP.

Но как production-основа он слаб:

- слишком много скрытой зависимости от in-memory state
- плохая воспроизводимость
- неочевидное поведение в distributed-сценариях

## 6. Какие решения лучше

### Вариант 1. PostgreSQL Full-Text Search + dense Qdrant

Рекомендованный практический вариант.

Идея:

- lexical retrieval перенести в PostgreSQL FTS
- dense retrieval оставить в Qdrant
- результаты сливать в приложении тем же RRF

Плюсы:

- детерминированность
- нет in-memory vocab drift
- естественно работает в нескольких инстансах
- меньше кастомной логики
- хорошо ложится на текущий стек

Минусы:

- это уже не «самописный BM25 sparse vector в Qdrant», а гибрид Postgres + Qdrant

### Вариант 2. Персистентный BM25 index per collection

Идея:

- хранить vocab, df и corpus stats в PostgreSQL
- version-ировать sparse schema per collection
- при изменении корпуса переиндексировать sparse-векторы

Плюсы:

- можно сохранить BM25-подобную модель
- поведение станет воспроизводимым

Минусы:

- заметно сложнее в реализации и эксплуатации
- нужно делать полноценный lifecycle индекса

### Вариант 3. Нормальный sparse model вместо самописного BM25

Идея:

- перейти на deterministic sparse encoder/model
- отказаться от собственного in-memory словаря

Плюсы:

- лучшее качество
- лучшее поведение в multi-instance среде
- меньше скрытой stateful-магии

Минусы:

- выше инфраструктурная сложность
- тяжелее inference

### Вариант 4. Dense-only + keyword fallback

Идея:

- убрать текущий sparse path
- оставить dense retrieval
- добавить простой keyword fallback на SQL/FTS

Плюсы:

- простая и надёжная система

Минусы:

- слабее lexical matching, чем у полноценного hybrid search

## 7. Что рекомендуется сделать дальше

Если выбирать прагматично, следующий лучший шаг такой:

1. перенести lexical retrieval в PostgreSQL FTS
2. оставить dense retrieval в Qdrant
3. сливать результаты через RRF
4. постепенно удалить текущий in-memory BM25 sparse path

Это даст самый хороший баланс между:

- качеством
- предсказуемостью
- простотой эксплуатации
- совместимостью с текущей архитектурой
